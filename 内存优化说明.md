# 内存优化说明

## 问题描述

当使用assignment-dependent模型时，如果实例规模较大（n值较大），协方差矩阵的大小为 `n² × n²`，会消耗大量内存。

例如：
- 如果 `n = 1000`，则 `n² = 1,000,000`
- 协方差矩阵需要 `1,000,000 × 1,000,000 = 1,000,000,000,000` 个元素
- 每个double占8字节，总共需要约 **8 TB** 内存

## 解决方案

### ✅ 方案1：自动延迟计算（已实现，推荐）

**程序已自动实现延迟计算模式！**

当协方差矩阵的内存需求超过 **50GB** 时，程序会自动切换到延迟计算模式：
- **不预先计算和存储整个协方差矩阵**
- **按需计算**协方差值（只在需要时计算）
- **大幅节省内存**（从TB级别降到MB级别）
- **代价**：计算时间会增加（因为需要重复计算）

**优点**：
- 无需手动配置
- 自动适应不同规模的实例
- 内存占用大幅降低

**缺点**：
- 计算时间会增加（因为需要重复计算协方差值）
- 对于非常大的实例，可能仍然较慢

### 方案2：增加JVM堆内存（适用于小到中等规模实例）

如果实例规模较小（内存需求 < 50GB），程序会使用正常模式（预先计算矩阵），此时可以增加堆内存：

#### 在IDEA中设置：

1. 打开 **Run** → **Edit Configurations...**
2. 选择你的运行配置（如 `DRTest1`）
3. 在 **VM options** 中添加：
   ```
   -Xmx16g
   ```
   其中 `16g` 表示最大堆内存为16GB，可根据需要调整

#### 在命令行中运行：

```bash
java -Xmx16g -cp ".:gurobi.jar:Jama-1.0.3.jar" DRTest1
```

#### 在Windows批处理文件中：

```batch
java -Xmx16g -cp ".;gurobi.jar;Jama-1.0.3.jar" DRTest1
```

### 方案3：使用较小的实例

如果延迟计算模式仍然太慢，考虑使用较小的测试实例。

## 内存需求估算

程序会在运行时自动计算并显示：
- 协方差矩阵的大小
- 所需内存（MB和GB）
- 当前JVM的最大堆内存
- 可用内存
- **是否使用延迟计算模式**

如果内存需求超过50GB，程序会自动切换到延迟计算模式，并显示：
```
内存需求过大 (XXX GB)，使用延迟计算模式（按需计算协方差）
注意: 这将增加计算时间，但可以节省大量内存
```

## 常见内存设置

| 实例规模 (n) | n² | 矩阵大小 | 内存需求 | 模式 |
|------------|----|---------|---------|------|
| 100 | 10,000 | 100M 元素 | ~800 MB | 正常模式 |
| 500 | 250,000 | 62.5B 元素 | ~500 GB | **延迟计算** |
| 1000 | 1,000,000 | 1T 元素 | ~8 TB | **延迟计算** |
| 2000 | 4,000,000 | 16T 元素 | ~128 TB | **延迟计算** |

**注意**：
- 延迟计算模式下，内存占用大幅降低（只存储原始数据和中间结果）
- 但计算时间会增加，因为需要重复计算协方差值
- 对于非常大的实例（如n>2000），可能需要考虑其他优化策略

## 技术细节

### 延迟计算实现

程序实现了以下优化：
1. **按需计算**：`getCovariance(idx1, idx2)` 方法在需要时才计算协方差值
2. **优化计算**：`computeQuadraticForm(v)` 和 `computeCovarianceMatrixVectorProduct(v)` 方法利用矩阵对称性减少计算量
3. **自动切换**：根据内存需求自动选择正常模式或延迟计算模式

### 性能影响

- **内存**：从TB级别降到MB级别（节省99%+内存）
- **时间**：计算时间会增加，具体取决于：
  - 实例规模（n值）
  - 场景数量（numAssignmentDependentScenarios）
  - 协方差矩阵的访问频率
